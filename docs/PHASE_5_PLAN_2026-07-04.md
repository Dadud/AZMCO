# Phase 5 Plan — DX11 draw path (resumed 2026-07-04)

## What's now confirmed (post-resume investigation)

### Calling convention (committed `ecf373d`)
- All 47 exports now use `WINAPI` (stdcall) in `Module.cxx`
- Verified via `dumpbin /exports`: 47/47 ordinals match EA's binary byte-for-byte
  (decorated names like `_THRASH_createwindow@16` are identical)
- mcity's `GetProcAddress("_THRASH_xxx@N")` lookups will resolve correctly

### mcity's HWND flow (reverse-engineered from upstream DX8 source)
- mcity calls `SelectState(25, hwnd_ptr)` to set the window handle
  (`RENDERER_MODULE_STATE_SELECT_WINDOW = 25` in `AZX/RendererModule.Basic.hxx:323`)
- `CreateGameWindow(width, height, format, options)` is "Never being called by the application"
  per upstream DX8 source comment — but the signature is real:
  `u32 STDCALLAPI CreateGameWindow(u32 width, u32 height, u32 format, u32 options)`
  Returns a window index (`State.Window.Count + MIN_WINDOW_INDEX`), NOT a BOOL.
- `SelectGameWindow(u32 indx)` chooses the active game window index
- `Lambdas.Lambdas.AcquireWindow()` is a callback mcity can register for cross-thread
  HWND acquisition; if non-NULL we should use it for our swap chain creation

### What this means for Phase 5 implementation
- `SelectState` must intercept state == 25 and store the HWND in `g_DX11.Window`
- When `CreateGameWindow` is called, allocate a real DX11 device + swap chain bound
  to `g_DX11.Window` (the HWND we got from SelectState)
- `SelectGameWindow` switches which texture is bound
- Returns: `u32`, not `BOOL` — adjust signature

## Phase 5 step-by-step plan

### Step 0 — DONE (commit ecf373d)
- Add WINAPI to all exports, complete .def decorations
- Verify exports match EA's binary

### Step 1 — HWND plumbing
1. Add a "window instance" array (up to 4 windows — see EA's MIN_WINDOW_INDEX behavior)
2. SelectState: intercept state==25, store HWND in the current window slot
3. CreateGameWindow: when HWND is available, create real DX11 device + swap chain
   bound to that HWND. Return window index (u32).
4. SelectGameWindow: switch current window slot

### Step 2 — Real device + swap chain in CreateGameWindow
Use `D3D11CreateDeviceAndSwapChain` with the HWND from SelectState.
This is what's currently in Init/CreateGameWindow stubs but with no HWND.
Should reuse the Phase 3 work that was already documented but never wired.

### Step 3 — Compile HLSL pass-through shaders at init
- `Source/R.DirectX.11.0.A/Shaders/PassThrough.hlsl` exists from Phase 3 doc
- Compile to DXBC via `D3DCompile()` from d3dcompiler.dll
- Create `ID3D11VertexShader` + `ID3D11PixelShader`
- Create vertex/index buffer pool with D3D11_USAGE_DYNAMIC + D3D11_MAP_WRITE_DISCARD
- Create input layout matching the HLSL VSInput struct

### Step 4 — First draw call: DrawRectangle
- Simplest of the 22 draw stubs
- Inputs: `(s32 x0, s32 y0, s32 x1, s32 y1, u32 color)`
- No state, no texture, no transform — just NDC vertices from pixel coords
- Map our vertex buffer, write 6 vertices (2 triangles), unmap
- Set primitive topology, bind shaders + RTV, draw, present

## Outstanding signature concerns (deferred until first smoke test)

The .def says `@4` for `AcquireState` but C++ declares `(u32, void*) = 8 bytes`.
EA's binary has `@4`, meaning EA's `AcquireState` takes only 4 bytes (one arg).
If smoke test reveals garbage data when mcity calls our state getter, we need to
fix the signature. For now, `return TRUE` masks the bug.

Same concern applies to:
- `SelectState(state, void*)` → def says `@8` (2 args = 8 bytes) — matches my sig ✓
- `AcquireGameWindowTexture(u32, void*)` → def says `@4` (1 arg = 4 bytes?)
- `SelectDevice(u32)` → def says `@4` ✓
- `SelectTexture(u32)` → def says `@4` ✓
- `SelectVideoMode(u32, u32, u32)` → def says `@12` (3 args = 12 bytes) ✓
- `AcquireState` mismatch noted — fix in next pass

These are all stub functions returning TRUE so the bug is silent for now.
